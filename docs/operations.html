รันและดูแลระบบ

ยกทั้ง stack ด้วยคำสั่งเดียว + ตรวจสุขภาพด้วย smoke test

สถาปัตยกรรม

ทุก request เข้าผ่าน Kong gateway (พอร์ต 8080) — Kong ตรวจ JWT (ปลั๊กอิน Lua jwt-hs512) แล้วฉีดเฮดเดอร์ X-Auth-User / X-Auth-Role ให้ service ปลายทางเชื่อถือ · แต่ละ service มีฐานข้อมูลของตัวเอง (DB-per-service)

          ┌─────────────┐
  Browser →│ Kong :8080  │→ auth :8081     ─ authdb
          │ (jwt-hs512) │→ catalog :8082  ─ catalogdb
          └─────────────┘→ order :8083    ─ orderdb   → (calls catalog: decrement stock)
                         → chat :8084     ─ chatdb    → (calls catalog / social)
                         → social :8085   ─ socialdb  → (calls catalog / chat)
Repoพอร์ตหน้าที่
marketplace-gateway8080Kong DB-less + ปลั๊กอิน jwt-hs512
marketplace-auth8081สมัคร/ล็อกอิน/JWT+refresh/roles
marketplace-catalog8082หมวดหมู่/ร้าน/สินค้า + ตัดสต็อกอะตอมมิก
marketplace-order8083ตะกร้า/checkout/ออเดอร์/สถานะ
marketplace-chat8084แชตเรียลไทม์ (REST + WebSocket)
marketplace-social8085FB gateway (webhook/connect/sync, mock)
marketplace-web3000/5173React 19 / Vite / TS
marketplace-common—ไลบรารีร่วม (JWT verify, header auth, RFC7807)
marketplace-deploy—docker-compose + run.sh + smoke.sh

Prerequisites

  • Docker + Docker Compose
  • JDK 21 & Maven (สำหรับ build jar) · Node.js 20+ (สำหรับ web dev)
  • clone repo marketplace-* ทั้งหมดไว้ในโฟลเดอร์แม่เดียวกัน

รัน stack (คำสั่งเดียว)

cd marketplace-deploy
cp .env.example .env          # ปรับ JWT_SECRET ให้ยาว ≥ 64 bytes ก่อนใช้จริง
./run.sh --build -d           # build jar ทุกตัว, render kong.yml, ยก stack
./smoke.sh                    # ตรวจ end-to-end 12 ข้อ ผ่าน Kong

เว็บ (โหมด dev): cd marketplace-web && npm run dev → http://localhost:5173

Kong แบบ DB-less อ่าน kong.yml ตอนสตาร์ทเท่านั้น — ถ้าแก้ route ต้อง docker compose up -d --force-recreate kong

Smoke test (12 ข้อ)

smoke.sh ทดสอบทั้งลูปผ่าน Kong จริง:

  1. health · 2) seller onboarding · 3) เปิดร้าน+สินค้า
  2. buyer checkout → สต็อกลด · 5) seller เห็นออเดอร์
  3. oversell: checkout ขนาน 2 อันบน stock=1 → สำเร็จ 1 / 409 อีก 1
  4. seller เลื่อนสถานะ · 8) guards (400/403)
  5. chat REST 401 + WS upgrade 101 + สร้างห้อง
  6. 2-client WS: buyer ส่ง → seller รับสด ผ่าน Kong
  7. FB inbound → โผล่ในกล่องแชต seller
  8. product sync → seller sync สินค้าไป FB catalog

Config (ตัวแปรหลักใน .env)

ตัวแปรความหมาย
JWT_SECRETคีย์ HS512 (≥ 64 bytes) — ต้องตรงกันทุก service + Kong
INTERNAL_API_KEYซีเคร็ตเรียกภายใน (order→catalog, social↔chat)
FB_VERIFY_TOKENโทเคน verify webhook ของ Meta
KONG_PROXY_PORTพอร์ตขาเข้า (ดีฟอลต์ 8080)
*_POSTGRES_*ชื่อ DB/user/pass ต่อ service
*_TARGETปลายทาง upstream ที่ Kong route ไป

Health checks

ทุก service มี GET /health → {"status":"UP"} · ผ่าน Kong: curl http://localhost:8080/health
